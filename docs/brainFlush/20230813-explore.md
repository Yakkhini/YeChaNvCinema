## 当我感到困惑

人类因困惑而思考，因思考而进步。我认为好奇心是一种优秀的品质，而幸运的是几乎每一个人都具备这一品质。但也令人遗憾的一点在于，不是每一个问题都具备答案。我们好奇生命的起源、时空的结构、宇宙的意义，但往往在寻找答案的路上交困。宇宙慷慨地赋予了我们存在性，又吝啬地藏着一切问题的终极答案。

所以我认为计算机科学是一门美妙的学科：人类既是计算机的缔造者，也是计算机的布道者。有关计算机的问题往往有人具备能力去解决，因为我们完全亲手构建起它的知识体系，设计它的运转规则。发达的互联网又让这些信息自然流淌，在不同人的脑中交换。

在学习编程、Linux 使用的过程中，我几乎每天都在遇到问题、解决问题。我希望可以分享我学到的经验。我享受这一过程，因为我相信自己在做一件美好的事——**寻找答案。**

### 各个信息的不同

在网上找答案并解决问题的过程可以理解成一个寻找、处理、利用互联网资料与信息的过程。但是不同的信息具有不同的特征，所以在解决问题之前我们最好能明白其中缘由。

1. 不同类型的信息专业性不同。比起闲聊群或论坛的内容，Issues 与问答（如 Stack Exchange）区的内容更专业。而如果你想找到更可靠的资料，往往只能去查阅官方文档（Documentation）和手册（Manual）。
2. 不同类型的信息可理解性也不同。一般来说，越专业且越长的信息越难理解，越随意且越短的信息越易理解，母语信息会比外文资料更容易看懂。
3. 信息具备时效性。在查阅相关资料时，最好看看正在查阅的资料是否有人在维护。维护良好的软件或组织发布的文档往往也维护良好，因而具备时效性。而问答帖却不会有人时时修补，即使它在彼时确实解决了一些问题。
4. 信息可以指向另一信息。在网上搜集资料时，可能回发现某一信息中提示了另一信源，如某篇回答中回答者贴出了相关文档，或者是某个 Issue 中有人 mention 了另一个 Issue。抑或是在这些资料中提到了一些未知的概念或者本以为与问题无关的概念，并表明这些概念其实是有影响、有用的。
5. 相同或相似特征的信息会聚集。网上有大大小小的论坛、社区或是 Wiki。不妨把这些信息载体称作寻找信息的「场所」。不难发现的是，在同一场所或同一类型场所中的信息也有一些相似特征，如其长度、专业性、主题，或是对于其他信源的引用。

在网上普遍会有人争论不同信息源的好与坏，或者建议不去哪个地方搜索答案。不过我持不同意见。在具体地论述这一争议前，不如先反问自己：我到底想要解决什么问题？

### 问题的特征

问题与答案如此相似，以至于它们就像一枚硬币的正反面。实际上我感觉问题与答案本身就是一体两面的姊妹，一个优秀清晰的问题本身就提示了通向答案的路途，而一个具备启发性的答案令人忍不住问出更多问题。

1. 不同的问题难度也不同。我想这一点是大家普遍同意的。有些问题天生很好解决，有些问题则需要进一步的探索。
2. 不同问题适合不同类型的答案。有一些问题可能答案很简短，比如「谁创始了 Linux？」——「Linus」或是「宇宙一切的终极答案是什么？」——「42」，也有的问题需要长篇大论才能解决，如「Linux 正则表达式如何使用？」的答案很可能是一篇 Manual。
3. 不同问题匹配不同的解决成本。如果花费一年的时间可以实现可民用核聚变，这对全人类来说都是一个天大的好消息。与之相比，如果一个学习编程者打算花费一整个月的时间来搞懂 `i++` 与 `++i` 的区别，那无疑是一种对时间的浪费。
4. 问题也可以分解或指向其他问题。如果不知道一个复杂困难的问题如何下手，不妨先盯着其中简单的部分做。在很多情况下完成一部分简单问题后又会有另一些不容易的分解问题转化为简单问题，就像做数独那样。同样的，可能会在解决问题时发现一些需要实现或解决的前置条件，也就是因一个问题而提出了另一个问题。当然，这也有可能会导致 [XY Problems。](https://xyproblem.info/)
5. 解决一个问题的思路或经验可以在类似问题中复用。在解决一个问题的过程中很可能会收获一些经验，比如去哪里可以找到有用的相关资料，或者是各个场所间信息特征的差异，甚至是相关的知识和概念。显然的，这些都是有用的财富，为下一次解决问题时提供一些初始的思路。在解决问题的过程中，人也得以学习和成长。

通过了解以上信息与问题的特征，我想一些额外的思考已经呼之欲出了。至少我认为 **提问与解决问题是一场效率的游戏。**为什么这么说？

### 从发现问题到解决问题

不妨假设现在我们正在解决一个遇到的问题。我希望可以把我的总结和经验以这样的顺序来讲解。

#### 真正搞懂问题

真的搞懂问题了吗？真的清楚、明白故障出自哪里吗？可以用一两句话来描述这个问题吗？有时候先提出另一个有价值的关于问题的问题是解决问题的好办法。在程序执行的过程中，往往关乎着很多很多组件的运行。这给了我们麻烦，却也给了我们从不同角度看问题的视角。一句代码的出错，或许是因语法错误而使得写的代码本身有问题，如错误地使用了指针或函数，也有可能是系统本身的环境变量、编译工具甚至是 LSP 的问题。

在清楚了问题后，需要明白该如何搜索想要的信息。更精确地说，该在搜索栏里敲下什么字。我自己的一个习惯是，尽量不要搜索连贯的句子，而是把想问的问题破碎成几个重要的关键字中间加空格搜索。尽管现在搜索引擎的算法已经越来越强大，但不可否认的是在搜索过程中字符匹配仍起到了不可代替的作用。一个句子可能有多种表达的方式，如果因为搜索内容对句式或无关单词表达的限定而错过了一些有用信息可太遗憾了。在很多情况下，「How Shell works?」都不如直接搜索「Shell」或者在知道可以去哪搜索的情况下「Shell Arch Wiki」更有用。注意，在大多数情况下或许都没必要把问号「？」打出来。搜索引擎知道你在问一个问题，而不是和搜索框闲聊。除此之外，掌握多级搜索的能力也是一个好办法。

?> 这也是为什么一个开发者需要具备一些英语水平会更好。

#### 去寻找最匹配的信息

有许多人持这样一种论调：如果你可以通过 Google、Stack Exchange 或是各种文档解决问题，就不要去 Baidu，CSDN 或者 Zhihu 去寻找答案。我大部分同意这一观点，实际上对于 80% 到 90% 的问题都应当去更可信的地方搜索。至少在计算机技术方面，英文世界比中文世界的积淀更多，具备更多有价值的资料。当然我们也在努力地提高中文相关文档的质量，比如写博客和读博客的你我。

可问题在于，真的需要每一次都要努力地找到最可信、最可靠、最权威的答案吗？当然，刨根问底是一个好习惯，但我们也要考虑 **问题的成本。**我认为每一个问题的场所乃至信息都具备独特的特征，也在适宜的情况下具备过人之处。以前文的 `i++` 和 `++i` 为例，我可能会在一段时间没有写某个语言的代码时遇到这种问题，我只是想要一点回忆而已。如果为这个问题还要去 Stack Exchange 或者 [cplusplus.com](https://cplusplus.com/)，[cppreference.com](https://zh.cppreference.com/w/) 去寻找就有点麻烦了。我更喜欢去菜鸟教程，甚至在知乎随手一搜，很快速地得到一个答案并解决问题。我不在乎得到一个二手或者多手信息，一个傻答案，因为我知道自己在问的就是一个傻问题。我唯一希望的就是它可以在几分钟内解决，把时间留给之后会遇到的需要深入探究的问题。

除此之外，国内的网站在遇到一些特定问题时甚至会比国外网站更有优势。我们知道国内的技术生态与国外不同，我们使用计算机遇到的问题有时候可能是国外社区很少人在意，甚至没人在乎的。比如中文 LaTeX 排版问题？WeChat 和 WeMeet 问题？这些是显而易见的。如果碰到某个看起来有些奇怪的问题，而在英文世界里搜集不到有用信息，有时候回到 Baidu 搜索一下可能会有意想不到的收获。

#### 算法课的启示

在前面我们讲，信息可以指向其他信息，问题也可以引出其他问题。于是以信息、问题为节点，以信息与信息，问题与问题之间的关系为链接而构成了两张图结构。这不禁让我联想起算法课遍历图的一些方法，**如深度优先搜索、广度优先搜索或是更普适的优先级搜索。**我倾向用它们类比搜索时的不同策略。

广度优先搜索代表着从不同角度看问题。比如一个问题与许多组件都相关时。这带来一些麻烦，也带来一些机会。当猜测问题出自不同的组件时，自然可能产生不同的关键词组。不妨依次搜索这些关键词，在比对之后或许就能排除一些方向，或是获得启发性的内容，又或者是进一步地广度搜索。要善于利用问题，**因为问题可以看作解决问题本身的元信息。**

而深度优先搜索则代表着 **多级搜索** 的技巧。多级搜索指的是根据搜索结果的启示而重新以新的关键词进行搜索，以期获得离答案更近的信息。这种方法可以修补对问题认识的偏差，或者是不合适的搜索关键词。同样的，这种方法也适合沿着一条线深入探究。很多时候会有人建议不要一股脑地把所有报错信息放搜索栏里搜索，包括我。不过我有时候会犯懒这么干，很多时候效果不好，甚至在 20% 到 30% 情况下搜索结果页面空白。接着就可以去除其中的无效甚至有害信息，比如我 Shell 的 Prompt，或者是某个我电脑上的路径，又或者是一些 Tui 符号，部分写的代码等等。这些具有特殊性的部分都会降低搜索结果的数量。然后再适当增减关键词，用空格切分不同部分。在结果中往往就会出现一些有用信息，顺着这些信息二次搜索很可能就会让整个搜索过程步入正轨。在遇到报错且根本不知道该从哪下手时，不妨用这个办法。

进一步讲，这也启发我们在顺着一条信息搜索链去寻找答案时遇到困难，不如回过头去搜索那些暂时还未选择去探索的其他信息，就像搜索过程的递归返回行为一样。最极端情况下甚至可以回头审视我们想要解决的问题是否恰当，以至于它在网上的相关内容较少。

当然，现实世界和算法毕竟还是有差异。有时候信息不会像数据结构那样离散，信息与信息间可能也有重复。同样的，我们不必像机器一样严格遵守算法，而可以依情况适当调整下一步搜索或者不搜索的内容、搜索的场所等。很多时候其实人会下意识地去继续打开搜索框或者某个网址，进行下一步的搜索。我喜欢把这背后的素养称作一种 **搜索直觉。**

#### 回顾和思考问题

在完成一次搜索与问题的解决后可以做一个回顾和思考，以让自己在这个过程中提升更多。不必很认真地专门腾出时间来做这件事，就像完成任务和作业一样。在搜索的过程中我就在不断地对自己搜索的过程做检查，或者是联系以前的搜索经验来解决问题。这就不断地、下意识地在完成回顾的过程。

1. 对于问题的认识是否有偏差

在解决问题结束后，很多时候会发现自己最开始问的问题有些「无厘头」，是自己对问题的认识不足，导致去寻找的方向有些没效率。为什么会这样？是我对代码的理解不够吗？我对这一出问题的实现背后运行的机理理解有误吗？我错误地理解了一些相关组件的作用吗？又或者是其他的一些原因。但不管怎样，这都会令我吸收教训，形成一种反馈条件，使我对于相关的事物理解更深一点，乃至在下一次问出更聪明些的问题。

2. 除了这一答案本身，还在求索的过程中学到了哪些相关知识

寻找答案的过程往往不是一帆风顺的，而是常常会在那些与问题相关却对解决问题没有帮助的信息流中漫游。如果仅仅把这些信息抛之脑后，那么这些漫游的时间也变成了被浪费 **（Wasted）** 的时间。但如果把这些相关知识填补在自己的知识框架里，或者是留个印象，就有可能对下次解决问题更有帮助，学习的效果也更好。

3. 对于信息源的积累

在搜索的过程中可能会在一些地方发现更有效率的信息，如同发现新大陆一样。比如某个 Wiki，某处 Manual，某个社区，某个文档。久而久之就会获得一处属于自己的「赛博地图」。如果把搜索栏比作共享单车或者步行的话，这些地方就像一个又一个快车站，引导搜索者快速奔向某些车站运营计划内的话题信息（意思是遇到某些问题去这里搜索可能会更有效率）。

4. 建立一种属于自己的「搜索直觉」

在以上的积累下，遇到问题或者搜索到某个节点信息时，可能搜索者灵机一动，产生一种「我或许应该立刻去 xxx 翻阅/寻找 xxx 信息，很有可能会有用」的感觉。我把这种感觉称作 **搜索直觉。**这并不是什么高深的超能力，而是每个人都可以拥有的赛博第六感，因为 **人天生就会搜索。**

相比于搜索，我在问答方面的经验不是很多。我认为这也是我的一个小缺点，也就是每次遇到问题时都执着于自己解决，而不是在适当的时候向别人提问。其实提问与回答是一件合理的事，优秀的问答可以增长每一位问答参与者的知识。另一方面来说，总有一些问题是还没有人问过的。如果那些对我有用的问答帖提问者羞于发帖提问，那么我也就不会看到那些优秀的答案。

对我来说，我认为在问答中隐藏着一个矛盾：如果我清楚地理解某个问题应该如何提问，那么我可能业已知道了该如何着手开始解决这个问题。也就是说，在我知道如何提一个正确的问题时，我不会第一选择去提问。但当我束手无策、不知道如何解决这一问题时，我对其认识往往也比较模糊，进而无法提出智慧的问题。

不过尽管如此，我也切实地参与了一些讨论，如 GitHub Issues 的问答。或者是我在各个交流群里旁观别人的一些讨论。因此我还是有一些基本的感悟。

### 不要每遇到不会的事就去提问

不少同学在日常的学习里养成这样一个习惯，每当遇到不会的问题时就去找那些更厉害的人虚心求教，或者是找师傅提问。这在许多行业里是一个优秀的品质，代表着他有着好学的性格。不过在计算机工程的世界里，大家更崇尚 **独立解决问题。** 解决问题的方法不仅来自抓来一个人提问，更来自于广博的互联网世界。一方面来说，遇到什么事都去问别人是走不下去的，因为未来将会遇到的问题浩如烟海，新技术的产生也将会带来无数新问题，如果知其然而不知其所以然就会在未来的路途中越走越难。另一方面来说，那些看起来懂很多的前辈也是从一无所知学起，他们不是天生理解能力更强，或者是脑袋里装了一个包罗万象的数据库，只是在反复锻炼中提升了自己解决问题的能力而已。

我遇到过不少这样什么都要问的同学，并且意识到他们并不是故意显得无礼，只是作为新人还没立起这种观念，不熟悉这些不成文的风土人情而已。清记得下次在解决一个问题时先尝试独立搜索，在提出一个问题时先检查有没有雷同问题。如果提问者把自己先前的努力以简洁的方式附出，那么往往会收到更多的宽容，即使他问了一个不太智慧的问题。因为大家知道提问者不是有意做伸手党，而是解决问题的能力还不足罢了。

### 适当宽容那些傻问题

如我前文所述的矛盾产生的一个结果就是，每当我提问时我有很多次都提出了一些不算智慧的问题。推己及人，我在遇到那些不合时宜的提问时也会去看他们是否其实是对自己的问题认识不清才搞错的。当然每个人的时间都是宝贵的，所以也不用手把手地帮他解决问题。不妨给出一些简短但具有启发性的回答：「我认为你应该去了解一下 xxx」，「你对 xxx 的理解出了问题」，「你没有问清，或许在此之前你可以了解一下 xxx」等等等等。这样既省了自己的时间，也给了提问者新的方向。作为提问者，在看到这样的回答后就应当意识到自己不该再追问下去了，因为大家觉得这是一个提问者完全可以自己解决的问题。除此之外，如果回答者决定忽视这些问题或者给了一些负面的批评也完全合理。所以提问者不必为这种事而内耗，不如去思考自己在解决问题时落下了什么东西。实际上如果提问者在下一次提出一个智慧的问题，大家也会去欣然参与。

### 积极回馈社区

在学习的过程中，我受到过不少前辈的指点，也在许多优质问答中被启发。所以在我成长后，遇到那些我能力范围内可以解决的问题，或者是一个我可以提出一个好问题的机会时，何乐而不为呢？这样回馈社区的行为可以启发其他后来者，或者帮助开发者提高作品的质量。

## 后记

总的来说，解决问题与问答的能力是一个开发者的软实力。提高的独立解决问题的能力，才能在计算机学习的路上走得更远；只有积极参与社区问答，遵守社区规范，在问答中彼此尊重，提高问答内容的价值，才能吸引更多同学加入社区，帮助更多人的同时让社区生态发展得更好。
